# P12.4 — Sidebar-Konsolidierung

Stand: 2026-04-21 · v2.3-dev

## Ausgeliefert

### Mode-basierte Tab-Sichtbarkeit
Die 6 Sidebar-Tabs sind jetzt per `data-mode` mit dem Planning-Mode verknüpft:

| Tab | `data-mode` | Room-Mode | Event-Mode |
|---|---|---|---|
| 📐 Räume (`ib-rooms`) | `room` | ✅ | — |
| 🛋️ Möbel (`ib-furn`) | `room` | ✅ | — |
| 🏗️ Bau (`ib-arch`) | `room` | ✅ | — |
| 🔒 Sicher (`ib-security`) | `room` | ✅ | — |
| ✏️ Eigene (`ib-custom`) | *(universal)* | ✅ | ✅ |
| 🎭 Veranst. (`ib-events`) | `event` | — | ✅ |

**Ergebnis:**
- Room-Mode: **5 Tabs** (Räume/Möbel/Bau/Sicher/Eigene)
- Event-Mode: **2 Tabs** (Veranst./Eigene)

### Auto-Tab-Switch bei Mode-Wechsel
Neuer Event-Listener in `index.html`: wenn der aktive Tab beim Mode-Wechsel ausgeblendet würde (z.B. `ib-events` beim Wechsel zu Room), wählt die App automatisch einen sinnvollen Default (Room → `ib-rooms`, Event → `ib-events`). Der User hat nie eine leere Sidebar.

## Nicht ausgeliefert (bewusst deferred)

### 3-Tab-Strukturierung pro Mode (user's design-intent)

Der User hat eine präzisere 3-Tab-Struktur pro Mode vorgegeben:

**Room-Mode (geplant):**
- Tab 1: Räume
- Tab 2: Möbel (mit Bau + Sicher als Accordion-Untergruppen)
- Tab 3: Eigene

**Event-Mode (geplant):**
- Tab 1: Flächen (Ground-Planes + Zonen)
- Tab 2: Messebau (Rückwände, Theken, Licht, Technik)
- Tab 3: Eigene

### Warum deferred

Die 3-Tab-Strukturierung erfordert substantielle DOM-Änderungen:
1. **Bau + Sicher** in **Möbel** als Accordion zu schachteln: Die derzeitige `renderLeft()`-Funktion schaltet zwischen `leftMode`-Strings um und rendert komplett unterschiedliche Inhalts-Bäume. Eine Accordion-Vereinigung braucht Refactoring des render-Dispatchers + neue UI-State für "welches Accordion offen".
2. **Event-Tab splitten** in "Flächen" vs. "Messebau": Das bedeutet `ib-events` → 2 neue Tabs, mit neuer `showLeft`-Dispatch-Logic und Migration des aktuellen `events`-Render-Pfads in zwei getrennte Render-Funktionen.
3. **Drag-Drop-Sources** in der Sidebar haben hartkodierte IDs und Selektoren — eine Tab-Restrukturierung muss jedes davon durchgehen.

Für einen autonomen Batch ohne E2E-Coverage der Drag-Drop-Flows ist das zu riskant. Der aktuelle PR löst das Kern-Problem (CSC-User sieht keine Event-Tabs mehr und umgekehrt) ohne Regressions-Risiko.

### Follow-up für v2.4

Separate, sorgfältige PR mit:
1. `showLeft()` refactoring zu einer Registry (`{tabId → {label, render, modes}}`)
2. Neue Tabs `ib-flachen` + `ib-messebau` als Event-Mode-Defaults
3. Bau + Sicher als `<details>`-Elemente innerhalb des Möbel-Tabs
4. Playwright-E2E-Test für jeden Drag-Drop-Pfad
5. Schritt-weise Migration (erst neue Tabs einführen, dann alte verstecken, dann entfernen)
