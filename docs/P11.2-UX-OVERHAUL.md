# P11.2 — UX-Overhaul (Menu-Restructure + Modal-Template + Touch-Targets)

Stand: 2026-04-21 · v2.2-dev

## Fokus dieser PR (ausgeliefert)

Diese PR konzentriert sich auf **nicht-destruktive** CSS- und Dokumentations-Arbeit. Die vollständige Menu-Neustrukturierung steht als kuratierter Plan — Umsetzung erfolgt in P11.4 und P11-Folgephasen, damit der Legacy-Code nicht bei Merge-Konflikten auseinanderfällt.

### Ausgeliefert
- **Touch-Targets 44×44 px** auf Mobile (`@media max-width:1024px`) — WCAG 2.1 AA-Konformität.
  - `.tbt, .mpill, .tbm-item, .mbt, .cmd-item, .mode-seg button`: `min-height:44px`.
  - `.icon-btn, .ib, .touch-target`: `min-height:44px, min-width:44px`.
- **`.modal-footer`-Utility-Klasse** — einheitlicher Footer-Pattern für alle zukünftigen Modals (rechts-ausgerichtete Button-Reihe, Top-Border).
- **`.touch-target`-Utility-Klasse** — kann an jedes winzige interaktive Element angehängt werden, das nachträglich vergrößert werden muss.
- **Menu-Struktur-Proposal** (unten) für iterativen Roll-Out.

### Defer nach v1.1 / nach P11-Batch
- Tatsächliche Menu-Re-Konstruktion der 155 Items (Risiko: onclick-Handler-Referenzen brechen bei Umordnung)
- Modal-Template-Refresh aller 48 Modals (sehr großer, hochriskanter Diff)
- Sidebar-Tab-Reduktion auf 5 Haupt-Tabs

**Begründung:** Die Legacy-`index.html` hat 21k Zeilen inline-JS mit hartkodierten DOM-Referenzen. Eine Big-Bang-Restrukturierung ohne vorherige DOM-Referenz-Extraktion (`querySelectorAll`-Audit) ist zu riskant für einen autonomen Batch. Die Restrukturierung wird nach der CSS-Ausgliederung (P11.3) sicherer.

## Modal-Template (einheitlich)

```html
<div class="mover" id="m-example">
  <div class="modal">
    <h2>Titel</h2>
    <div class="modal-body" style="max-height:60vh;overflow-y:auto">
      <!-- Body-Content -->
    </div>
    <div class="modal-footer">
      <button class="mbt" onclick="closeM('m-example')">Abbrechen</button>
      <button class="mbt ok" onclick="...">OK</button>
    </div>
  </div>
</div>
```

Alle 48 Modals auf dieses Schema bringen: Header (h2), scrollbarer Body, rechts-ausgerichteter Footer. Das wird iterativ Modal-für-Modal nach P11.3 umgesetzt, damit der Diff pro PR überschaubar bleibt.

## Touch-Target-Audit

| Element-Typ | Vorher | Nachher (mobile) |
|---|---|---|
| `.tbt` | 22 px | 44 px |
| `.mbt` | 29 px | 44 px |
| `.tbm-item` | 26 px | 44 px |
| `.cmd-item` | 30 px | 44 px |
| `.mode-seg button` | 28 px | 44 px |
| `.icon-btn` | 28 px | 44 px (auch Breite) |
| `.ib` | 46 × 48 | 46 × 48 (bereits OK) |

Desktop (`>1024 px`): behält die dichte Density — nur Mobile/Touch bekommt die Vergrößerung via Media-Query.

## Sidebar-Tab-Reduktion

**Aktuell (links):** ~8–10 Tabs je nach Mode, dicht gedrängt.  
**Ziel:** 5 Tabs max.

Vorgeschlagene Konsolidierung:
1. 📦 **Katalog** (war: Möbel, Bau, Sicherheit, Pflanzen, Messe) — Sub-Kategorien als Accordion
2. 🏠 **Räume** (war: Räume + Flächen)
3. 🎨 **Material** (war: Material + Image-Upload)
4. 🌿 **Eigene** (war: Meine Templates + Community Marketplace)
5. ℹ **Info** (war: Statistik + Legende + Meta)

Umsetzung nach P11.3 (CSS-Extraktion), da die Sidebar-Struktur massiv mit inline-Event-Handlern verwoben ist.
