# P10.5 — Progressive Disclosure (3-Tier UI-Mode)

Stand: 2026-04-20 · CSC-Raumplaner v1.0+

## Konzept

Der Planner hat 155 Menu-Items (siehe P10.3). Für einen Neu-User ist das überwältigend. Lösung: drei UI-Modi, die sichtbare Features progressiv enthüllen.

| Modus | Icon | Zielgruppe | Sichtbar |
|---|---|---|---|
| **Einfach** | 🌱 | Neu-Verein, erster Versuch | Nur Basics: Räume zeichnen, Objekte platzieren, speichern, PDF-Druck |
| **Standard** | ⚖ | Gründungsphase, normaler Betrieb | + Behörden-Paket, Kostenvoranschlag, Compliance-Checks, Vorlagen, KI |
| **Profi** | 🚀 | Architekt/BIM-Workflow | + IFC/BIM-Export, Flucht-Pläne, Security-Reports, Team-Management |

## Technische Umsetzung

### 1. CSS-Regeln (Body-Attribut steuert Sichtbarkeit)

```css
body[data-ui-mode="simple"] [data-tier="pro"],
body[data-ui-mode="simple"] [data-tier="standard"] { display: none !important; }
body[data-ui-mode="standard"] [data-tier="pro"] { display: none !important; }
```

Default: `data-ui-mode="standard"` (der "richtige" Modus für 80 % der CSCs).

### 2. Tagging-Konvention

Jedes Element, das nicht im Einfach-Modus sichtbar sein soll, bekommt `data-tier="standard"` oder `data-tier="pro"`. Fehlendes Attribut = sichtbar in allen Modi.

Aktuell (P10.5 Init-Batch) getaggt:
- `data-tier="pro"`: IFC-Export, Flucht- & Rettungsplan, Sicherheitsbericht PDF
- `data-tier="standard"`: Behörden-Paket (.zip), Kostenvoranschlag PDF

### 3. UI-Mode-Selector (Topbar)

```html
<select id="ui-mode-select" onchange="setUIMode(this.value)">
  <option value="simple">🌱 Einfach</option>
  <option value="standard" selected>⚖ Standard</option>
  <option value="pro">🚀 Profi</option>
</select>
```

### 4. Persistenz

`localStorage.setItem('csc-ui-mode', '<mode>')`. Bootstrap läuft *vor* dem ersten Render (inline `<script>` vor `<body>`-Content), damit keine Feature-Items kurzzeitig aufblitzen.

### 5. API

Globale Funktion `window.setUIMode(mode: 'simple'|'standard'|'pro')` — kann aus anderen Skripten, der Command-Palette (P10.6) oder dem Onboarding-Flow aufgerufen werden.

## Rollout-Plan

### Phase 1 (jetzt) — Infrastruktur
- ✅ CSS-Regeln + Attribut-System
- ✅ UI-Selector in Topbar
- ✅ Persistenz via localStorage
- ✅ 5 Top-Menu-Items initial getaggt (IFC, Evacuation, Security, Behörden, Kosten)

### Phase 2 (v1.1) — Full-Coverage
- Alle 155 Menu-Items reviewen und taggen
- Modal-Inhalte ebenfalls taggen (z.B. BIM-spezifische Sections im Export-Modal)
- Sidepanel-Toolbar-Buttons taggen
- Onboarding-Flow fragt "Welcher Modus passt zu dir?" beim ersten Start

### Phase 3 (v1.2) — Smart-Disclosure
- Nutzungs-Signale: wenn User 3× IFC-Export anfragt → Suggest "⏩ Zu Profi wechseln"
- Tooltip-Hint bei versteckten Features: "Aktiviere Profi-Modus, um BIM zu sehen"

## Edge-Cases

- **Keyboard-Shortcuts:** Shortcuts arbeiten tier-unabhängig. Ein Einfach-User, der Ctrl+Shift+I drückt, bekommt weiterhin IFC-Export — Shortcuts sind bewusste Profi-Eingriffe.
- **Deeplinks:** `?template=...` + `?project=...` funktionieren in allen Modi. Wenn ein Template Pro-Features referenziert, erscheint Hinweis.
- **KI-Agent:** Die Command-Palette (P10.6) zeigt *alle* Kommandos, unabhängig vom Modus — wer explizit sucht, bekommt. UI-Disclosure schützt Neu-User, nicht Suchende.

## Erfolg-Metriken

Nach Rollout tracken:
- Verteilung simple/standard/pro (Plausible-Event beim Mode-Change)
- Bounce-Rate pro Modus (Einfach-Modus sollte niedrigere Bounce haben)
- Feature-Discovery: wie lange bis ein Einfach-User in Standard wechselt?
