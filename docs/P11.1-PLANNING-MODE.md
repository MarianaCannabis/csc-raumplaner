# P11.1 — Planning-Mode-Switcher

Stand: 2026-04-21 · v2.2-dev

## Konzept

Zwei Arbeits-Modi in der Topbar. Segmented Control zwischen `🏪 Raumplanung` (Default) und `🎪 Veranstaltungs-Planung`. Persistiert in `localStorage.csc-planning-mode`.

```
┌───────────────────┐┌─────────────────────────────┐
│ 🏪 Raumplanung     ││ 🎪 Veranstaltungs-Planung   │
└───────────────────┘└─────────────────────────────┘
```

## Funktions-Erhalt-Garantie

Mode blendet aus, **löscht nichts**. Alle Buttons/Regeln bleiben im DOM bzw. in der Registry — sie sind nur per CSS / `isRuleActive()` gefiltert. Über die Command-Palette (Ctrl+K) und den Profi-UI-Mode (P10.5) sind Features in jedem Planning-Mode erreichbar.

## Technisches Modul

`src/modes/planningMode.ts` exportiert:
- `currentMode(): PlanningMode` — liest aus localStorage, Default `'room'`
- `setMode(m)` — persistiert + setzt `body.dataset.planningMode` + feuert `csc-mode-change`
- `onModeChange(cb)` — Event-Abo, gibt Unsubscriber zurück
- `isRuleActive(ruleModes, active?)` — Filter-Helper für Registry

Tests: `src/modes/planningMode.test.ts` — 6 Cases (default, persist, event-fire, rule-filter).

## CSS-Selektoren

```css
body[data-planning-mode="room"]  [data-mode="event"] { display:none !important; }
body[data-planning-mode="event"] [data-mode="room"]  { display:none !important; }
```

Elemente **ohne** `data-mode` sind in **beiden** Modi sichtbar (Default-Universal).

## Compliance-Engine

Das `Rule`-Interface bekam ein optionales `modes: ('room'|'event')[]` Feld. `registry.evaluateAll()` und `evaluateForRoom()` filtern jetzt via `listActiveRules()` nach dem aktiven Mode. Regeln ohne `modes` laufen immer (back-compat mit den 14 bestehenden Regeln, die nicht getaggt wurden).

### Mode-Zuordnung der 21 Regeln

**Nur `room`:**
- `memberLimit` (KCanG §11)
- `preventionOfficer` (KCanG §23)
- `visualScreen` (KCanG §14)
- `youthProtection` (KCanG 18+)
- `poiCscDistance` (KCanG §13)
- `ausgabe` (CSC-spezifische Raumstruktur)
- `sozial` (CSC-Lounge)

**Nur `event`:**
- `messeHeightLimit` (Messe-Höhenbegrenzung)

**Beide (untagged, universal):**
- Alarm, Eingang-Sec, Fluchtweg, Feuerlöscher, Kamera, Kapazität, Lager, Notausgang, Rauchmelder, Room-Door-Width, Room-Min-Area, Room-Smoke-Detector, WC *(13 Regeln)*

Grund: Fluchtweg/Brandschutz/Türbreite/WC/Feuerlöscher sind unabhängig vom Nutzungs-Kontext verpflichtend.

## Confirm-Dialog

Beim Mode-Wechsel: `window.confirm()`:

> Planung-Modus wechseln?
>
> Aktuelle Objekte bleiben erhalten, aber manche Regeln/Kategorien werden ausgeblendet.

Nicht unterbrechend bei `cur === new` (idempotent).

## Edge-Cases

- **Import-Order:** `planningMode.ts` setzt das Body-Attribut im Modul-Scope, damit die CSS-Regeln vor dem ersten Paint greifen. Eine zweite Inline-Bootstrap-Sequenz in `index.html` läuft VOR dem TS-Bundle, damit auch bei schlankem Supabase-Login-Gate kein Flicker auftritt.
- **Private-Mode:** `localStorage.setItem` kann fehlschlagen — `try/catch` erhält Default-Mode `'room'`.
- **Rule-Modes empty array:** `modes: []` läuft auch immer (wie `undefined`) — bewusst tolerant.
