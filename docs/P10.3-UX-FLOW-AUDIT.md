# P10.3 — UX-Flow-Audit

Stand: 2026-04-20 · Review der 5 Kern-User-Journeys im CSC-Raumplaner.

## Methode

Statisches UI-Inventory (kein Headless-Browser):
- **48** Modals (`#m-*`) inline in `index.html`
- **155** Topbar-Menu-Items (`.tbm-item`) verteilt auf die 4 Hauptmenüs (Datei / Bearbeiten / Ansicht / Extras)
- **63** `.tbt`-Toolbar-Buttons (Sidepanels + schwebende Leisten)

Click-Zählung: minimaler Pfad vom App-Start bis Ziel-Zustand. Kein Walking-Skeleton-Shortcut (kein Keyboard-Shortcut eingesetzt), damit die reale Maus-Pfad-Länge sichtbar wird.

---

## Flow 1 — Neues Projekt anlegen + speichern

| # | Aktion | Target |
|---|---|---|
| 1 | Datei-Menü öffnen | Topbar `📁 Datei` |
| 2 | "🆕 Neues Projekt" | Modal `m-welcome` |
| 3 | Projekt-Name eingeben | Input-Feld |
| 4 | "🏁 Starten" | Submit |
| 5 | Datei-Menü öffnen (wieder) | Topbar |
| 6 | "💾 Speichern" | Cloud-Save (falls eingeloggt) |

**Klicks:** 6 (wenn Login aktiv) / **8** wenn `m-auth`-Magic-Link-Flow zuerst läuft.

**Pain-Points:**
- Save-Button ist nicht prominent — liegt im Submenü, nicht als Primary-Action in der Topbar.
- Nach "Neues Projekt" kein impliziter Auto-Save → User verliert bei Refresh ohne Warnung.

**Empfehlung P10.4:** Primary-Save-Button in Topbar (Cmd+S ist schon da via P8, aber nicht sichtbar dokumentiert).

---

## Flow 2 — Raum zeichnen + Objekt platzieren

| # | Aktion | Target |
|---|---|---|
| 1 | Linkes Sidepanel → Tool-Reiter | default: aktiv |
| 2 | "🏠 Raum zeichnen"-Button | Tool aktivieren |
| 3–4 | 2× Canvas-Klick (Start/End) | Rechteck ziehen |
| 5 | Optional: Raum umbenennen | Modal `m-rename` |
| 6 | Katalog öffnen | Sidepanel-Tab |
| 7 | Kategorie aufklappen | Accordion |
| 8 | Objekt auswählen | Button |
| 9 | Canvas-Klick | Platziert Objekt |

**Klicks:** 9 Happy-Path, 13 inkl. Rename.

**Pain-Points:**
- Kein Preview beim Hover über Katalog-Objekt → User weiß vor Platzierung nicht, wie das Teil aussieht.
- Raum zeichnen erfordert exakten Click-Drag-Release; Cancel-Geste (ESC) nicht überall verdrahtet.
- Katalog hat 6 Top-Level-Kategorien mit je 10–30 Items ⇒ Scroll-Overhead.

**Empfehlung P10.4/P10.5:** 3D-Preview-Tooltip im Katalog-Card (lazy); ESC-Global-Cancel im Tool-State-Machine.

## Flow 3 — Projekt in die Cloud speichern + von anderem Gerät laden

| # | Aktion | Target |
|---|---|---|
| 1 | `m-auth` öffnen | Topbar 👤-Icon |
| 2 | E-Mail eingeben + Magic-Link | Input + Button |
| 3 | Externes Mail-Client → Link klicken | Browser-Redirect |
| 4 | Datei → Speichern | Cloud-Save |
| 5 | Anderes Gerät: Schritte 1–3 | — |
| 6 | Datei → "📂 Projekte laden" | `m-load` |
| 7 | Projekt-Eintrag anklicken | Deserialisieren |

**Klicks:** 7 Intra-App + 1 Email-Link-Klick.

**Pain-Points:**
- Magic-Link-UX: User muss Tab-Switch machen. Kein Polling / auto-reload im aktiven Tab.
- "Projekte laden"-Modal zeigt rohe Timestamps statt "vor 2 Stunden".

**Empfehlung:** Pseudo-Relative-Timestamps im Load-Modal (+10 LOC, hoher UX-Win).

## Flow 4 — Export (DXF/PDF/IFC)

| # | Aktion | Target |
|---|---|---|
| 1 | Datei-Menü | Topbar |
| 2 | Submenu "📤 Export" aufklappen | — |
| 3 | Format-Item anklicken | Download-Trigger |

**Klicks:** 3 (optimal).

**Pain-Points:**
- Keine Export-Preview (ausser PDF → öffnet Druckansicht). DXF und IFC blind heruntergeladen.
- IFC-Export ist in Flow-Hierarchie verschachtelt — für BIM-Workflow-User schmerzhaft.

**Empfehlung:** Export-Modal mit Format-Cards + Settings (z.B. "nur Erdgeschoss").

---

## Flow 5 — Template laden + anpassen (Mari-Jane-Onboarding)

| # | Aktion | Target |
|---|---|---|
| 1 | Datei → Vorlagen | `m-templates` |
| 2 | Tab "System" | Default |
| 3 | Template-Card klick | `applyTemplate(id)` — überschreibt aktives Projekt |
| 4 | Confirm-Dialog OK | JS-confirm |
| 5 | Räume anpassen | pro Raum: Doppelklick → `m-room` → Size ändern → Schließen |

**Klicks:** 4 bis Template geladen, +3 pro angepasstem Raum.

**Pain-Points:**
- `JS-confirm()` ist modal-blocking + ugly-native — kein Auto-Backup vor `applyTemplate`.
- Tab-Bar (System/Meine/Community) versteckt — User bemerkt nicht, dass eigene Templates da wären.

**Empfehlung:** `confirm()` → app-internes Modal mit "Projekt-Backup erstellen"-Checkbox (default on).

---

## Cross-Flow-Patterns

| Pattern | Vorkommen | Bewertung |
|---|---|---|
| Topbar-Menü → Submenu → Action | 155× `.tbm-item` | Konsistent — gut |
| Sidepanel-Toolbar-Button | 63× `.tbt` | Gut, aber Icon-only-Buttons brauchen bessere Hover-Tooltips (siehe P8.6-aria-labels — erledigt) |
| Modal mit inline-Body statt eigener Komponente | 48 Modals | Legacy-Muster, in v2.1 als Web-Components kapseln |
| Context-Menü im Canvas (rechter Mausklick) | ~5 Stellen | Nicht discoverable — User weiß ohne Hilfe-Modal nicht davon |

## Zusammenfassung

| Flow | Klicks Happy-Path | Hauptbremser |
|---|---|---|
| 1. Neues Projekt | 6 | Save versteckt im Submenü |
| 2. Raum + Objekt | 9 | Kein Katalog-Preview |
| 3. Cloud-Sync | 7 | Magic-Link-Context-Switch |
| 4. Export | 3 | Keine Preview/Settings |
| 5. Template | 4 | Native-confirm, kein Auto-Backup |

**Nord-Stern:** Für 90 % der Sessions sollten Flow 1+2 unter 8 Klicks bleiben. Aktuell: 15 (Flow 1+2 ohne Keyboard). Keyboard-Shortcuts + Command-Palette (P10.6) reduzieren das auf ~4.

**Fortführung:** P10.4 (UI-Konsolidierung) adressiert Doppel-Buttons und Icon-Inkonsistenz; P10.5 (Progressive Disclosure) blendet Expert-Features per Default aus; P10.6 (Command-Palette) verkürzt Power-User-Pfade auf 2 Klicks.
